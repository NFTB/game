# 开发计划

## 项目目标

先做出一个可联机游玩的多人竞拍 MVP，再根据测试反馈扩展角色、藏品、经济系统和表现效果。

第一版核心闭环：

```text
游客登录 -> 大厅 -> 创建/加入房间 -> 准备 -> 竞拍 -> 成交 -> 结算 -> 返回大厅
```

客户端负责展示和交互，服务端负责房间状态、出价校验、成交判定和结算。

## 技术范围

- 客户端：Unity 6000.4.7f1 + C#
- 服务端：Go
- 实时通信：WebSocket
- 数据库：PostgreSQL 或 MySQL，MVP 前确定
- 缓存：Redis，联机 MVP 后接入
- 配置：JSON，先放在 `shared/config`
- 协议：JSON 消息，先放在 `shared/protocol`
- 特效：核心玩法稳定后再加

## 开发周期预估

以下周期统一按“个人 + AI 辅助开发”估算。AI 可以加速代码生成、文档整理、问题排查和重构，但玩法验证、Unity 场景调试、美术取舍、联机测试仍需要人工反复确认。

整体预估：

- 可玩本地 Demo：2 到 4 周
- 联机 MVP：7 到 10 周
- Alpha 测试版：3 到 5 个月
- 较完整测试版：5 到 8 个月

当前建议目标：先用 7 到 10 周完成联机 MVP。

## 阶段 1：玩法规则和本地 Demo

目标：不依赖服务端，先验证竞拍玩法是否成立。

预计周期：2 到 3 周。

主要任务：

- 明确单局人数：建议 2 到 4 人
- 明确单局轮数：建议 MVP 先做 5 轮
- 明确初始金币：建议 MVP 先用 1000
- 明确出价方式：暗拍或限时提交
- 明确成交规则：最高出价者获得藏品并扣除金币
- 明确藏品估价规则：真实价值、估值区间、误差范围
- 明确胜负规则：最终资产价值或净收益最高者获胜
- Unity 制作本地房间界面
- Unity 制作竞拍界面
- Unity 制作简单结算界面
- 用假玩家模拟出价

MVP 暂不做：

- 角色技能
- 复杂道具
- 排行榜
- 好友系统
- 付费系统
- 大量特效

验收标准：

- 单机模式能跑完一整局
- 玩家能看懂当前藏品、估价、倒计时、自己的金币和出价结果
- 一局结束后能看到每个玩家的收益和排名

## 阶段 2：服务端核心状态机

目标：让 Go 服务端能独立管理一局竞拍。

预计周期：2 到 3 周。

主要任务：

- 设计玩家会话结构
- 设计房间结构
- 设计房间阶段：大厅、准备、竞拍、结算
- 设计竞拍回合状态机
- 实现创建房间
- 实现加入房间
- 实现离开房间
- 实现准备和取消准备
- 实现开局逻辑
- 实现发放藏品
- 实现出价校验
- 实现成交判定
- 实现单局结算
- 增加基础日志

验收标准：

- 服务端单测能覆盖房间和竞拍状态机
- 非法出价会被拒绝
- 重复出价、超时出价、金币不足出价都有明确处理
- 服务端能在没有 Unity 客户端的情况下完成一局模拟

## 阶段 3：WebSocket 联机 MVP

目标：Unity 客户端接入 Go 服务端，真实玩家可以联机跑完整闭环。

预计周期：3 到 4 周。

客户端任务：

- 实现 WebSocket 连接
- 实现游客登录
- 实现大厅界面
- 实现创建房间
- 实现加入房间
- 实现房间玩家列表
- 实现准备按钮
- 实现服务端状态同步
- 实现出价提交
- 实现成交结果展示
- 实现结算展示
- 实现基础错误提示

服务端任务：

- 实现 WebSocket 网关
- 实现消息 envelope 解析
- 实现 `auth.guest`
- 实现 `room.create`
- 实现 `room.join`
- 实现 `room.leave`
- 实现 `room.ready`
- 实现 `auction.bid`
- 实现 `room.snapshot` 广播
- 实现 `auction.round_started` 广播
- 实现 `auction.bid_accepted` / `auction.bid_rejected`
- 实现 `auction.round_settled`
- 实现 `auction.finished`

验收标准：

- 两个客户端能进入同一房间
- 所有玩家准备后能开局
- 每轮出价能同步到服务端
- 成交结果所有客户端一致
- 结算结果所有客户端一致
- 玩家中途断开不会让服务端崩溃

## 阶段 4：数据持久化和基础经济

目标：把临时内存数据逐步落库，形成可持续测试的账号和资产系统。

预计周期：2 到 3 周。

主要任务：

- 确定数据库：PostgreSQL 或 MySQL
- 设计玩家表
- 设计藏品表
- 设计玩家藏品表
- 设计对局记录表
- 设计金币流水表
- 增加数据库迁移脚本
- 增加游客账号持久化
- 增加金币结算落库
- 增加藏品获得记录
- 接入 Redis 缓存在线状态和房间短期状态

验收标准：

- 玩家退出后再次登录仍能读取账号数据
- 对局结束后金币和藏品能正确保存
- 经济变动有流水记录
- 服务端重启后不会丢失玩家长期资产

## 阶段 5：核心内容扩展

目标：从“能玩”扩展到“有变化、有策略、有重复游玩动力”。

预计周期：3 到 5 周。

主要任务：

- 扩展藏品配置到 30 到 50 个
- 增加藏品稀有度
- 增加估价线索
- 增加少量角色，建议先做 3 到 5 个
- 增加少量技能，优先做容易验证的被动或回合技能
- 增加收藏柜
- 增加简单战绩
- 增加每日奖励，可选

验收标准：

- 不同藏品有明显价值差异
- 玩家每轮有信息判断空间
- 角色技能不会破坏服务端权威判定
- 配置修改后客户端和服务端能保持一致

## 阶段 6：体验打磨

目标：提升界面反馈、节奏感和产品完成度。

预计周期：2 到 4 周，可与内容扩展部分并行。

主要任务：

- 统一 UI 风格
- 增加按钮反馈
- 增加倒计时表现
- 增加卡片翻转或揭示动画
- 增加成交音效
- 增加出价音效
- 增加结算动画
- 增加轻量粒子效果
- 增加稀有藏品高亮
- 优化移动端或 PC 端布局

暂缓：

- 复杂 Shader
- 大规模 3D 展厅

验收标准：

- 玩家能快速理解自己当前该做什么
- 关键操作有明确反馈
- 倒计时和成交瞬间有紧张感
- UI 在目标分辨率下不遮挡、不溢出

## MVP 任务拆分

客户端：

- `AppBootstrap`：客户端启动入口
- `RealtimeClient`：WebSocket 连接和消息收发
- `AuctionSession`：客户端会话状态
- `LobbyView`：大厅界面
- `RoomView`：房间界面
- `AuctionView`：竞拍界面
- `SettlementView`：结算界面

服务端：

- `realtime`：WebSocket 连接管理
- `game`：房间、玩家、竞拍状态机
- `httpapi`：健康检查和普通 HTTP 接口
- `store`：数据库访问接口
- `config`：环境变量和配置读取

共享：

- `shared/config/game_rules.json`：玩法规则
- `shared/config/items.json`：藏品配置
- `shared/protocol/messages.json`：消息名列表

## 风险和应对

联机状态复杂：

- 先做服务端权威状态机。
- 客户端只展示服务端快照。

规则不好玩：

- 先做本地 Demo。
- 不急着堆内容和美术。

Unity 包版本不兼容：

- 先保持最小依赖。
- 后续通过 Unity Package Manager 逐个添加。

配置前后端不一致：

- 配置先统一放在 `shared/config`。
- 服务端校验关键规则。

开发范围膨胀：

- MVP 只保留登录、房间、竞拍、结算。
- 其他系统进入 Alpha 后再做。
